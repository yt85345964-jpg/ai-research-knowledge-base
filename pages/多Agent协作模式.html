多Agent协作模式

核心定义

**一句话定位**：多个 AI Agent 通过明确的协作规则和通信机制共同完成复杂任务的设计范式。

为什么重要

单个 Agent 受限于能力边界和上下文窗口，复杂任务往往需要多个专业 Agent 协同。理解不同的协作模式，是设计和评估 Agent 系统的关键。

对我们团队的启示：这个模式可以直接指导我们设计 AI 知识库的编译流程。WorkBuddy 作为"主理人"调度处理脚本、格式转换、Wiki 编译等"子Agent"，通过 AGENTS.md 固化的就是协作规则。

关键机制

机制概述

1. 调度-执行（编排者模式）：WorkBuddy 专家团采用的模式，也是最基础的多Agent结构。


主 Agent 分析用户请求 → 拆解为子任务
    → 为每个子任务生成目标、上下文和输出要求
    → 子 Agent 在独立上下文中执行
    → 主 Agent 收集中转结果 → 冲突处理、信息合并 → 最终回复

关键规则：

  • - 主理人只管调度，不代写专业产出
  • - 成员之间不直接通信，所有信息流经主理人中转
  • - 成员专业产出以成员结论为准

2. 编排-执行（Harness 模式）：OpenClaw + Harness 架构采用的模式——编排层与执行层通过 ACP(Agent Client Protocol) 桥接。


编排层(OpenClaw Gateway) 分析任务 → 路由决策
    → 简单问答: Pi(内置 Harness) 直接处理
    → 代码任务: 通过 ACP 拉起 Claude Code 在独立 worktree 中工作
    → 研究任务: Research 子会话 + web search
    → 执行结果回传编排层 → 按审批策略(自动/复核/人工)提交 → 状态同步

关键特点：

  • - 编排层管「何时做、给谁做」，执行层管「具体怎么写代码」
  • - 会话可恢复（重启不丢进度）、独立 worktree 隔离（多任务并行不冲突）
  • - 分级审批：直接执行 / 委托编排层复核 / 人工审批 / 自动开 PR
    • 来源：素材笔记/AI技术/OpenClaw-Harness智能工作管理架构方案

3. 其他协作范式

模式 特点 适用场景
辩论-仲裁 多个Agent辩论，仲裁者决定 需要多方观点的决策
层级委派 任务层层分解下发 大型复杂项目
共享黑板 所有Agent读写共享状态 需要实时状态同步

关键设计要点

  • - **任务分配预检**：在派发任务前先做能力匹配，防止误派。步骤：解析用户意图 → 对照能力速查表 → 能力匹配则派发，不匹配则告知限制。
  • - **消息中转的必要性**：避免 Agent 之间互相调用造成的上下文污染；所有通信有记录，所有决策有责任人；出错时可追溯。
  • - **异常处理**：超时机制（如超过10分钟未回传 → 通报用户）；失败重试上限（如失败超2次 → 终止并说明原因）；强制性交付检查清单。
  • - **预设 Workflow**：将常见协作场景固化为 Workflow，每个包含：触发条件、执行阶段、关键决策点、产物交付规范。
