多Agent协作模式

一句话定义

多个 AI Agent 通过明确的协作规则和通信机制共同完成复杂任务的设计范式。

为什么重要

单个 Agent 受限于能力边界和上下文窗口，复杂任务往往需要多个专业 Agent 协同。理解不同的协作模式，是设计和评估 Agent 系统的关键。

核心模式

1. 调度-执行（编排者模式）

WorkBuddy 专家团采用的模式，也是最基础的多Agent结构。

工作流程：


主 Agent 分析用户请求 → 拆解为子任务
    → 为每个子任务生成目标、上下文和输出要求
    → 子 Agent 在独立上下文中执行
    → 主 Agent 收集中转结果 → 冲突处理、信息合并 → 最终回复

关键规则：

  • - 主理人只管调度，不代写专业产出
  • - 成员之间不直接通信，所有信息流经主理人中转
  • - 成员专业产出以成员结论为准

2. 其他协作范式

模式 特点 适用场景
辩论-仲裁 多个Agent辩论，仲裁者决定 需要多方观点的决策
层级委派 任务层层分解下发 大型复杂项目
共享黑板 所有Agent读写共享状态 需要实时状态同步

关键设计要点

任务分配预检

在派发任务前先做能力匹配，防止误派：

  1. 解析用户意图
  2. 对照能力速查表
  3. 能力匹配 → 派发；不匹配 → 告知限制

消息中转的必要性

  • - 避免 Agent 之间互相调用造成的上下文污染
  • - 所有通信有记录，所有决策有责任人
  • - 出错时可追溯

异常处理

  • - 超时机制（如超过10分钟未回传 → 通报用户）
  • - 失败重试上限（如失败超2次 → 终止并说明原因）
  • - 强制性交付检查清单

预设 Workflow

将常见协作场景固化为 Workflow，每个包含：

  • - 触发条件
  • - 执行阶段
  • - 关键决策点
  • - 产物交付规范

对我们团队的启示

这个模式可以直接指导我们设计 AI 知识库的编译流程。WorkBuddy 作为"主理人"调度处理脚本、格式转换、Wiki 编译等"子Agent"，通过 AGENTS.md 固化的就是协作规则。

相关概念

  • - Agent提示词设计模式 — 单个Agent的结构化设计
  • - WorkBuddy — 该模式的产品化实现
  • - Dify — 未来将支持多 Agent 协同的 LLMOps 平台
  • - AI原生组织 — YC 内部的 Dream Cycle = 多 Agent 协作 + 自我改进循环

行业趋势

多个产品平台正在将多 Agent 协作作为未来发展方向：

  • - WorkBuddy：已产品化的专家团（调度-执行模式）
  • - Dify：正在规划多智能体协作系统（不同专长 Agent 配合完成复杂任务，来源 素材笔记/AI 产品/2026-07-02_Dify平台Agent开发入门指南）
  • - 共同趋势：从单 Agent 走向 Agent 团队，从手动编排走向智能调度
综合判断：多 Agent 协作正在从"实验性设计模式"进入"产品化标准功能"阶段。
