rtk

核心定义

**一句话定位**：高性能 CLI 代理，在命令输出到达 LLM 之前过滤压缩，降低 60-90% Token 消耗。

核心功能

是什么

rtk 是一个单文件 Rust 二进制程序，零外部依赖。它作为 CLI 代理运行在 Shell 和 AI Agent 之间：拦截常见开发命令的输出，在传给 LLM 之前进行智能过滤、分组、截断和去重，将 Token 消耗降低 60-90%，额外开销 <10ms。

68.1k Stars（2026-07），支持 100+ 命令，兼容 Claude Code、Copilot、Cursor、Codex 等 14 种 AI 编程工具。

核心价值

给 AI 编程 Agent 的上下文窗口"瘦身"。Agent 执行 `git status`、`cargo test` 等命令时，大部分输出是 LLM 不需要的噪音。rtk 在中间层过滤掉冗余，让 Agent 用更少 Token 获得同样有效的信息。

一个 30 分钟 Claude Code 会话中，原始命令输出约 118,000 Token，经过 rtk 后降至约 23,900 Token（节省 80%）。

工作原理


Agent --命令--> Shell --原始输出--> LLM     ← 传统模式（大量 Token）
Agent --命令--> rtk --过滤--> LLM            ← rtk 模式（压缩输出）

四种策略：

  • - **智能过滤**：去除注释、空白、样板代码
  • - **分组聚合**：按目录聚合文件、按类型聚合错误
  • - **截断**：保留关键上下文、裁掉冗余行
  • - **去重**：重复日志行折叠为计数

关键特性

  • - **自动重写钩子（Auto-Rewrite Hook）**：安装钩子后，所有 Bash 命令自动被重写为 rtk 等价命令（`git status` → `rtk git status` 等），实现 100% rtk 采用率。⚠️ 注意：钩子仅对 Bash 工具调用生效，Claude Code 的 Read、Grep、Glob 等内置工具不经过钩子。
  • - **覆盖范围**：
类别 代表命令 典型节省
文件操作 `ls`、`read`、`find`、`grep` 70-80%
Git `status`、`diff`、`log`、`push` 75-92%
测试 `pytest`、`cargo test`、`go test`、`jest` 90%
构建检查 `cargo build`、`eslint`、`tsc` 80%
容器 `docker ps/logs`、`kubectl` 80%
包管理 `pnpm list`、`pip list`、`bundle install` 60-80%
  • - **分析能力**：
    • 
rtk gain             # Token 节省统计
rtk gain --graph     # 30 天趋势图
rtk discover         # 发现遗漏的优化机会
rtk session          # 会话级 rtk 采用率

适用场景

适用 不适用
AI 编程 Agent（Claude Code/Cursor/Copilot/Codex 等）的上下文瘦身 Windows 原生环境（钩子依赖 Unix Shell）
高频 Git/测试/构建命令的 Token 压缩 需要覆盖预定义命令集之外自定义脚本的场景
30 分钟会话可节省 80% Token 的大型项目 对输出完整性要求极高、不能容忍偶发过滤的场景

团队评估

rtk 作为 CLI 代理在 Shell 与 AI Agent 之间压缩命令输出，可显著降低 Token 消耗（60-90%）。其与 Codex 可叠加作为 Token 优化层；WorkBuddy 同样面临上下文窗口效率问题，rtk 思路可参考。注意 Windows 原生支持有限，建议用 WSL。

局限性

  1. **Windows 原生支持有限**：钩子依赖 Unix Shell，Windows 上自动重写不可用，只能退化为 CLAUDE.md 注入模式。建议用 WSL。
  2. **不覆盖所有命令**：只支持预定义的命令集，自定义脚本需要自行配置。
  3. **可能过滤掉有用信息**：偶尔会过于激进，导致 LLM 缺少关键上下文。可配置 `tee` 模式在失败时保存完整原始输出。
