RAG（检索增强生成）

核心定义

**一句话定位**：RAG（Retrieval-Augmented Generation，检索增强生成）是一种"检索式知识库"范式——查询时从原始文档检索相关片段 → LLM 理解 → 综合生成回答，让模型在生成时"带着资料作答"，而非仅依赖参数内记忆。

为什么重要

RAG 是当前 LLM + 外部知识的标准方案之一：它让模型能回答训练数据之外、或需要最新/私有资料的问题，覆盖面广、上手快、无需预先综合。对团队而言，RAG 是 LLM Wiki（编译式知识库）的互补对照物——理解两者差异，才能在"检索"与"编译"之间做出正确的架构取舍。

关键机制

机制概述

查询时实时检索：把用户问题向量化 → 在向量数据库中做语义检索 → 取回最相关的文档片段 → 连同问题一起喂给 LLM → LLM 综合生成回答。知识不沉淀在模型或中间层，每次查询都重新"翻书"。

Karpathy 指出传统 RAG 的致命缺陷："There's no accumulation."（没有积累）——每次查询都从原始文档重新检索、理解、综合，是重复计算，知识不会随使用而复利沉淀（来源：LLM Wiki）。

编译 vs 检索

范式 核心动作 类比 知识是否沉淀
**传统 RAG（检索式）** 查询时从原始文档检索→理解→综合 解释型语言：每次运行都重新编译 ❌ 不沉淀，重复计算
**[[LLM Wiki]]（编译式）** 摄入时 LLM 主动提取、整合、更新 Wiki 编译型语言：提前编译成中间表示 ✅ 一次编译，多次复用

关键差异：RAG 像「每次考试都重新翻书」；LLM Wiki 像「提前上过课，知识点已经记住，考试直接答」。

适用场景

  • - 大量未精细阅读的文档：RAG 覆盖面广。
  • - 简单事实检索：速度快，不需要预综合。
  • - 与 LLM Wiki 互补：生产环境常先查 Wiki（精确、综合好、人类可读），再查向量数据库（大规模、语义检索），合并结果生成回答。
知识类型 推荐方案 原因
精心阅读过的文档 [[LLM Wiki]] 已预综合，质量高
大量未精细阅读的文档 RAG 覆盖面广
需要跨文档综合的分析 [[LLM Wiki]] 综合已预编译在 Wiki 中
简单事实检索 RAG 速度快，不需要综合
