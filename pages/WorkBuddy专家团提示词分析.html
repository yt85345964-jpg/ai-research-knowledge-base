WorkBuddy专家团提示词全曝光：多Agent协作原来是这样产品化的

来源：杨添的知识库 → 微信公众号文章

核心结论

WorkBuddy的专家 = 封装好的单个Agent

WorkBuddy的专家团 = 封装好的多Agent协作流程

专家的提示词结构（通用模板）

从「用户体验架构师」和「长文档手稿专家」两个专家中提炼出通用结构：

  1. **角色覆盖和身份锚定**（Role Override）：让模型不被之前上下文影响，重置为专家身份
  2. **核心使命**：定义专家的职责范围
  3. **关键规则和默认行为**：约束Agent不要自由发挥（如 Foundation-First 原则）
  4. **工作流程和执行步骤**：明确的SOP，分步执行
  5. **交互模板和质量标准**：交付物定义、成功标准（Success Metrics）、沟通风格示例
  6. **能力支持和工具声明**：Agent可用的工具和技能
  7. **持续牵引与下一步引导**：完成后引导用户下一步
  8. **Agent Runtime 说明**：记忆系统、安全规则、任务管理等

本质上 = 普通Agent + 领域方法论 + 交付模板 + 工作流约束

专家团的设计（多Agent协作）

核心模式：调度-执行模式（编排者模式）

主理人设计

  • - 主理人不直接干活，只管调度：拆需求、配成员、看进度、收产出
  • - 提示词示例：「你是AI内容创作专家团的创意制片人司远（Soren），负责协调团队...」

团队工作区

  • - 主理人创建工作区，统一管理产出物
  • - 任务完成后删除工作区

团队架构

  • - 每个成员有明确的 Agent ID（系统用）和中文名字（用户看）
  • - 系统通过 Agent ID 调度

任务分配预检

  • - 主理人在派发任务前必须先做能力匹配
  • - 提示词中有明确速查表，列出常见误排场景
  • - 流程：解析意图 → 对照速查表 → 能力匹配则派发，不匹配则告知限制

预设 Workflow

  • - 内容创作专家团预设8个Workflow
  • - 每个Workflow定义：触发条件、执行阶段、关键决策点、产物交付规范

团队协作规则

  1. 建立团队：主理人亲自创建
  2. 调度成员：按SOP阶段拉成员入协作
  3. 消息中转：所有跨成员信息流经主理人中转
  4. 成员结论为准：专业产出必须由对应成员输出后再采信

为什么用中转模式：避免Agent之间互相调用造成的上下文污染和追溯困难。

异常处理

  • - 成员超过10分钟未回传 → 通报用户
  • - 同一任务失败超2次 → 终止并说明原因
  • - 强制性交付检查清单

多Agent协作范式

  • - 调度-执行（WorkBuddy采用的模式）
  • - 辩论-仲裁
  • - 层级委派
  • - 共享黑board
