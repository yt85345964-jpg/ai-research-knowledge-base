Agent提示词设计模式

一句话定义

将专业领域的专家行为模式固化为 Agent 提示词的通用结构模板，使 Agent 从"自由发挥"变为"可预期的专业交付"。

为什么重要

普通 Agent 缺乏领域约束时容易自由发挥、输出不稳定。通过结构化的提示词设计，可以让 Agent 像真正的专家一样按照固定方法论工作，产出质量和风格更加可控。

核心原理

八层结构

从 WorkBuddy 专家的提示词设计中提炼出的通用结构：

  1. **角色覆盖和身份锚定**
  2. - 使用 Role Override 机制清空之前上下文中的角色
  3. - 定义 Role、Personality、Memory、Experience
  4. - 关键：让模型不被之前对话中的身份影响
  1. **核心使命**
  2. - 一句话定义专家的职责范围
  3. - 明确专家的能力边界（能做什么、不做什么）
  1. **关键规则和默认行为**
  2. - 约束 Agent 不要依赖通用知识自由发挥
  3. - 示例：Foundation-First（先打地基再开发）
  4. - 示例：Developer Productivity Focus（替开发者做架构决策）
  1. **工作流程和执行步骤**
  2. - 明确的 SOP，分步拆解任务
  3. - 每步有清晰的输入、输出和检查点
  4. - 示例：分析项目需求 → 创建技术基础 → UX结构规划 → 开发者交接文档
  1. **交互模板和质量标准**
  2. - 定义交付物模板（如 Theme Toggle 组件代码）
  3. - 定义 Success Metrics（可验证的成功标准）
  4. - 定义 Communication Style（沟通风格示例）
  1. **能力支持和工具声明**
  2. - Agent 可用工具和技能清单
  3. - 工具调用方式和限制说明
  1. **持续牵引与下一步引导**
  2. - 任务完成后引导用户下一步操作
  3. - 保持对话的连续性和生产力
  1. **Agent Runtime 说明**
  2. - 记忆系统、安全规则、工作模式
  3. - 任务管理、上下文管理、工具使用规则

本质公式


专家Agent = 普通Agent + 领域方法论 + 交付模板 + 工作流约束
        ≈ 普通Agent + 领域知识Skills

推理模型 vs 通用模型的策略差异

这是提示词工程的一个重要进阶分支（来源：素材笔记/AI 技术/2026-07-02_DeepSeek从入门到精通）：

维度 推理模型（R1/o3） 通用模型（GPT-4o/V3）
提示策略 **简洁指令**，聚焦目标 **结构化引导**，显式拆解步骤
原则 "要什么直接说" "缺什么补什么"
陷阱 角色扮演、过度拆解会干扰逻辑 直接提复杂推理可能跳过步骤

对八层结构的启示：

  • - 对推理模型使用八层结构时，**第 4 层（工作流）应简化**——模型已内化推理，过度拆解反而限制其能力
  • - 对通用模型仍建议完整使用八层结构，特别是工作流拆解和交互模板
综合判断：推理模型更接近"专家"——它有内在方法，你只需要明确目标；通用模型更接近"助手"——你需要补上它缺失的结构。

框架对比：八层 vs 三段式

不同平台的提示词设计理念各有侧重（来源：素材笔记/AI 产品/2026-07-02_Dify平台Agent开发入门指南）：

维度 WorkBuddy 八层结构 Dify 三段式
复杂度 高（全面覆盖交付全链路） 低（聚焦核心三要素）
适用场景 产品化固化的专家角色 开发者灵活定制的 Agent
结构 角色→使命→规则→流程→模板→工具→引导→运行时 角色 → 流程 → 约束
核心理念 "把专家经验固化为可复用的模板" "最小必要结构，给开发者留空间"
综合判断：两者不是对错之分，而是场景之分。八层适合"封装好给人用"，三段式适合"搭好框架让人改"。本知识库的 AGENTS.md 更接近八层——因为知识库运营是一个需要固化交付标准的专业行为。

主流实现

  • - WorkBuddy — 专家系统实现
  • - Custom GPTs（OpenAI）— 概念相似但结构简化
  • - Coze Bot — 字节跳动的Agent平台

与 Dream Cycle 的闭环

YC 内部的技能自我进化（Dream Cycle）为提示词设计模式提供了企业级闭环：


探索发现有效模式 → 固化为技能（提示词设计模式） → Dream Cycle 持续优化 → 再探索

这说明提示词设计不是终点，而是中间态——好的提示词应该在使用中持续进化，而不是一次性写完就锁死。
