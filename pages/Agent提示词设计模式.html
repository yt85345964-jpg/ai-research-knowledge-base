Agent提示词设计模式

核心定义

**一句话定位**：将专业领域的专家行为模式固化为 Agent 提示词的通用结构模板，使 Agent 从"自由发挥"变为"可预期的专业交付"。

为什么重要

普通 Agent 缺乏领域约束时容易自由发挥、输出不稳定。通过结构化的提示词设计，可以让 Agent 像真正的专家一样按照固定方法论工作，产出质量和风格更加可控。

关键机制

机制概述

八层结构（从 WorkBuddy 专家的提示词设计中提炼出的通用结构）：

  1. **角色覆盖和身份锚定**：使用 Role Override 机制清空之前上下文中的角色；定义 Role、Personality、Memory、Experience；关键：让模型不被之前对话中的身份影响。
  2. **核心使命**：一句话定义专家的职责范围；明确专家的能力边界（能做什么、不做什么）。
  3. **关键规则和默认行为**：约束 Agent 不要依赖通用知识自由发挥；示例：Foundation-First（先打地基再开发）、Developer Productivity Focus（替开发者做架构决策）。
  4. **工作流程和执行步骤**：明确的 SOP，分步拆解任务；每步有清晰的输入、输出和检查点；示例：分析项目需求 → 创建技术基础 → UX结构规划 → 开发者交接文档。
  5. **交互模板和质量标准**：定义交付物模板（如 Theme Toggle 组件代码）；定义 Success Metrics（可验证的成功标准）；定义 Communication Style（沟通风格示例）。
  6. **能力支持和工具声明**：Agent 可用工具和技能清单；工具调用方式和限制说明。
  7. **持续牵引与下一步引导**：任务完成后引导用户下一步操作；保持对话的连续性和生产力。
  8. **Agent Runtime 说明**：记忆系统、安全规则、工作模式；任务管理、上下文管理、工具使用规则。

推理模型 vs 通用模型的策略差异（来源：素材笔记/AI技术/DeepSeek从入门到精通）：

维度 推理模型（R1/o3） 通用模型（GPT-4o/V3）
提示策略 **简洁指令**，聚焦目标 **结构化引导**，显式拆解步骤
原则 "要什么直接说" "缺什么补什么"
陷阱 角色扮演、过度拆解会干扰逻辑 直接提复杂推理可能跳过步骤

对八层结构的启示：

  • - 对推理模型使用八层结构时，**第 4 层（工作流）应简化**——模型已内化推理，过度拆解反而限制其能力。
  • - 对通用模型仍建议完整使用八层结构，特别是工作流拆解和交互模板。
综合判断：推理模型更接近"专家"——它有内在方法，你只需要明确目标；通用模型更接近"助手"——你需要补上它缺失的结构。

框架对比：八层 vs 三段式（来源：素材笔记/AI产品/Dify平台Agent开发入门指南）：

维度 WorkBuddy 八层结构 Dify 三段式
复杂度 高（全面覆盖交付全链路） 低（聚焦核心三要素）
适用场景 产品化固化的专家角色 开发者灵活定制的 Agent
结构 角色→使命→规则→流程→模板→工具→引导→运行时 角色 → 流程 → 约束
核心理念 "把专家经验固化为可复用的模板" "最小必要结构，给开发者留空间"
综合判断：两者不是对错之分，而是场景之分。八层适合"封装好给人用"，三段式适合"搭好框架让人改"。本知识库的 AGENTS.md 更接近八层——因为知识库运营是一个需要固化交付标准的专业行为。

渐进式披露（Progressive Disclosure）（Claude Code 提出的核心设计哲学，来源：素材笔记/AI技术/Claude_Code设计哲学渐进式披露）：

**不一次性把所有信息塞给模型，而是给模型工具让它自己层层深入**——借鉴自 UI 设计领域「只在需要时展示复杂信息」的原则。
  • - **是什么**：渐进式披露是 Agent Skills 系统中的上下文加载策略，将技能的**可发现性元数据**与**完整执行指令**分层存储、按需加载。
  • - **解决什么问题**：100 个技能全量加载需 ~500,000 tokens，超出上下文限制。
  • - **怎么用**：通过 `SKILL.md` 分离 YAML Frontmatter（元数据层）和 Markdown 正文（指令层），加外部资源文件（资源层），构成三层按需加载架构。

Agent Skills 三层披露：

层级 内容 加载时机
第一层：元数据 SKILL.md 的 YAML frontmatter（name + description） 启动时全部加载到 system prompt
第二层：核心指令 SKILL.md 的正文 Markdown Claude 认为相关时按需加载
第三层：详细资源 引用的 `reference.md` / `forms.md` 等附属文件 Claude 主动点击进入时加载

对八层结构的启示：

  • - 第 4 层（工作流程）应**按需展开**——不要把整个工作流的每一步都硬塞进初始提示词。
  • - 第 6 层（能力支持）应**元数据化**——只暴露 name + description，详细使用方式由 subagent 按需加载。
  • - 配套使用 `claude-code-guide` subagent 处理低频问题，避免主上下文被低频信息污染。

经典反例 vs 正例：

反例 正例
一次性把 100 个技能完整指令塞进 system prompt 启动只加载 100 个技能的元数据（name + description）
把"如何添加 MCP"完整文档放在主提示词 触发 `claude-code-guide` subagent 按需读取
给 Claude 越多越好 **克制 = 释放 AI 潜力**

与 Dream Cycle 的闭环（来源：素材笔记/AI组织/YC内部AI实操）：

YC 内部的技能自我进化（Dream Cycle）为提示词设计模式提供了企业级闭环：


探索发现有效模式 → 固化为技能（提示词设计模式） → Dream Cycle 持续优化 → 再探索

这说明提示词设计不是终点，而是中间态——好的提示词应该在使用中持续进化，而不是一次性写完就锁死。详见 AI原生组织。
